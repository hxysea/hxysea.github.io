19.状态模式

State Pattern

  1. What is it?

    根据某种状态采取对应的行动，即是状态设计模式的基本思想。但是其精髓在于将状态(State)抽象出来，不同的状态作不同的实现，再抽象出来一个用于存储当前状态的对象(Context),且Context实现了State接口。所以请求流向如下：
    Client—>Request—>Context—>State#doAction()

  2. How implement it?

    • 实现State Interface
    • 实现Concrete State Implement
    • 实现Context

  3. Code Demo

    A. State Interface

    B. Concrete State Implementations

    D. Test Program

  4. Important Point

    • 使用了面向对象的多态特性，使实现更加清晰
    • 减少出错率，更易扩展新行为，且与客户端松耦合
    • 程序扩展更灵活，更健壮，更易维护
    • 避免if/else和switch/case的使用，减少耦合

18.命令行模式

Command Pattern

  1. What is it?

    在请求->响应模型中，将不同请求抽象成一个命令接口，不同需求去实现这个命令接口，然后定义一个接收者接口来执行实际需求的操作，这样就把请求与实际处理解耦，进一步将命令接口隔离，由调用者接口去调用命令接口方法。至此，即为命令行模式的核心思想。请求流向为：
    Client—->Request—->Invoker—->Command—->Receiver

  2. How to implement it?
    • 设计Receiver接口，执行具体操作
    • 根据需求实现Receiver接口
    • 设计Command接口
    • 根据需求实现Command接口
    • 设计Invoker类

  3. Code Demo

    A. Receiver Classes

    B. Command Interface and Implementations

    C. Invoker Class

    D.Test

  4. JDK Example

    • java.lang.Runnable#run
    • javax.swing.Action#…
  5. Important Points
    • 命令接口定义了实现规范
    • 命令实现与接受者实现分离开
    • 一个需求方法对应一个命令实现，命令接口在invoker与recevier之间建立了桥梁
    • invoker类只是向前传递请求
    • 客户端负责创建comand对象和receiver对象，并将其关联
    • 客户端负责创建invoker对象，并将其与command对象关联
    • 易于扩展，直接在receiver对象增加方法，增加新方法的command实现即可
    • 方法太多，则命令对象太多

17.策略模式

Stategy Pattern

  1. What is it?

    所谓策略就是方法，在程序里面就映射为算法。策略模式的精髓就是对一个具体的任务有几种不同的算法，将这些算法抽象出一个公共的接口，客户端根据不同的需求选择不同的实现，

  2. How to implement it?
    • 设计算法接口类
    • 实现不同算法类
    • 设计算法操作对象
    • 设计应用场景类

  3. Code Demo

  4. JDK Example

    • java.util.Collecions#sort(List list, Comparator<? super T> c)
    • java.util.Arrays#sort()
  5. Important Point
    • 将对某一具体任务的处理算法抽象出一个接口，客户端使用更加灵活且解耦
    • 与状态模式相仿，但是状态模式利用的是context对象的不同状态而策略模式是根据客户端传递的参数选择不同的实现，殊途同归。

16.观察者模式

Obeserver Pattern

  1. What is it?

    订报纸是观察者模式在生活中一个很形象的例子，首先用户向报社预定报纸，报社每出一个新版本都会邮寄给订阅用户。简单来说，观察者模式可以这样定义：一个主题对象，一群关注该主题对象的观察者对象，主题对象更新就通知所有关注该主题的观察者对象。

15.责任链模式

Chain of Responsibility Pattern

  1. What is it?

    编码中的try/catch中的catch链则是责任链模式的一个典型应用，ATM取款则是生活中责任链模式思想的一个经典体现。如果需求中需要根据不同的输入选择一串不同处理方式，则这些处理方式可以构成一个处理链。简而言之，责任链模式的精髓在于将不同处理逻辑通过链式的方式组合在一起，从而达到解耦的目的。

  2. How to implement it?
    • 设计待处理对象
    • 设计责任链接口
    • 设计责任链实现

  3. Code Demo

    A. Base Classes and Interface

    B. Concrete Chain Implementations

    C. Creating the Chain

    D. Test the Chain

  4. JDK Example

    • javax.servlet.Filter#doFilter()
    • java.util.logging.Logger#log()
  5. Important Points
    • 与客户端解耦，客户端无需关注具体的处理方式
    • 责任链中的对象可以处理请求对象的全部或者部分
    • 设计时要关注责任链中各对象的处理方式，避免链中出现不可能执行到的对象，以及最后一个对象的异常处理机制
    • 责任链模式需要再可维护性与实现类中做个折中，以达到解耦的目的

14.中介者模式

Mediator Pattern

  1. What is it?
    • 中介的概念在生活中很普遍，比如租房中介。租房中介在租客与房东之间建立了联系，租客通过中介与房东取得了联系，房东通过中介寻找租客。
    • 在企业应用中，如果多个对象之间需要通信，如果是对象与对象之间直接交互和通信，造成的结果就是多个对象之间紧耦合在一起。此时，引入中介的概念则是明智之举。通俗来说，就是创建一个对象充当中介的角色，对象通过中介通信，而不是对象直接通信。
    • 飞机调度中心充当也是一个中介的角色，飞行员与调度中心通信，调度中心为其规划合适路线，调度中心再通知其它飞机。

  2. How to implement it?
    群组聊天的例子实现：

    • 定义中介者接口，定义通信方法
    • 定义需要通信的接口
    • 定义中介者实现
    • 定义通信实现

  3. Code Demo
    A. Mediator Interface

    B. Colleague Interface

    C. Concrete Mediator

    C. Concrete Colleague

    D. Mediator Pattern Client

  4. JDK Example

    • java.util.scheduleXXX（）方法
    • java.util.concurrent.execute()方法
    • java.lang.reflect.Method.invoke()方法
  5. Important Points
    • 中介者模式适用于多个对象之间通信复杂的情况，使用中介对象实现通信逻辑，从而达到解耦和易扩展的功能
    • JMS使用中介者模式结合观察者模式来实现应用订阅和发布数据给其他数据的功能
    • 如果仅仅为了解耦使用中介者模式会造成中介者越来越多，也越来越难维护。此种情况，不建议使用中介者模式

13.模板方法模式

Template Method Pattern

  1. What is it?
    举个例子：

    • 任务：建造房屋

    • 建造方法：

      1. 打地基
      2. 打木桩
      3. 造墙
      4. 造窗子
      5. 封顶
      6. 装修

      如建造房屋的例子，1-6顺序不可变，每一步根据需求自行选择具体方法，比如可以做木墙也可以做土墙，但是必须先造墙，再造窗子，顺序不可变。
      如果经过抽象出来的对象包含一类方法，这类方法是一套算法并且这套算法每一步的执行顺序不可变，则可以使用模板方法模式。换句话说，模板方法的核心思想就是封装一套任务执行的步骤且每一步执行顺序固定，不可更换。子类可以根据需求，自定义每一步的具体实现。

  2. How to implement it?
    • 设计抽象模板方法类（包含模板方法和基本实现以及需要子类实现的抽象方法）
    • 实现模板类（实现抽象方法或子类需求中需要覆写的方法）

  3. Code Demo
    A. Template Method Abstract Class

    B. Template Method Concrete Classes

    C. Template Method Pattern Client

  4. JDK Example

    • InputStream/OutputStream/Reader/Writer类的非抽象方法
    • AbstractList/AbstractSet/AbstractMap类的非抽象方法

  5. Important Points

    • 模板方法中的步骤顺序固定，模板方法声明为final，子类和基类每一步方法实现可不同
    • 超类选择子类方法加载
    • 基类中的默认实现就像一个钩子可以被子类覆写，也可声明为final不被子类覆写

Behavioral Design Patterns

行为型设计模式

对象之间进行通信或者说是进行交互是构建一个系统必不可少的环节。将对象之间交互的一些优秀思想进行总结提炼就构成了行为型设计模式的精髓，换言之，行为型设计模式为系统设计中对象交互提供了一套优秀的解决方案，这套方案使对象之间松耦合且更容易实现灵活的扩展。

其中，代表性的设计模式有如下几个：

  1. Template Method Pattern
  2. Mediator Pattern
  3. Chain of Responsibility Pattern
  4. Observer Pattern
  5. Strategy Pattern
  6. Command Pattern
  7. State Pattern
  8. Visitor Pattern
  9. Interpreter Pattern
  10. Iterator Pattern
  11. Memento Pattern

行为型设计模式关注的是对象之间交互进行设计，其中的精髓需要不断的理解及实践放得其真谛。最后，结合创建型以及结构型设计模式共同创建一个健壮的系统。

12.装饰者模式

Decorator Pattern

  1. What is it?

    所谓装饰者模式即在原有功能基础之上增加一些新功能，本质上使用组合的方式来完成需求功能。以往设计使用继承的方式在基础功能上增加新功能，那就有个弊端如果同时需要增加多个功能则要继承多个类，使新类与原有逻辑耦合在一起。如果抽象出一层装饰类，再组合一个基础接口成员变量，则完成了装饰的雏形。以基础接口为主线依次继承装饰类，使用组合方式取代继承方式是装饰者模式的核心思想。

  2. How to implement it?
    • 设计基础接口
    • 实现基本功能
    • 实现父装饰功能
    • 实现子装饰功能

  3. Code Demo

  4. Important Points

    • 根据需求，运行时动态扩功能，选择多时（根据基本功能扩展需求较多）更易扩展和维护
    • 缺点是会创建很多相似的装饰者子类
    • java.io.*中的BufferedReader等

11.桥接模式

Bridge Pattern

  1. What is it?

    一个类的功能需要实现多个类则意味着与这些类都耦合，如果将这些类按照其层级关系进行适当的组合即将继承关系转换为组合，这就是桥接模式思想的来源。简而言之，从实现中解耦出抽象层，以组合的方式实现功能。

  2. How to implement it?
    • 分析类图关系，解耦出抽象层
    • 将此抽象层组合到父类中
    • 继承或实现父类
